# Prompt para Agente: Reorganizacion de Dotfiles con .hidden

## Contexto

Tengo un flake NixOS en `/home/necro/nixFlake` que gestiona dos hosts:
- **desktop** — PC de sobremesa, AMD, NVIDIA, btrfs pendiente
- **notebook** — ThinkPad, Intel

Home Manager gestiona mi usuario `necro` en ambos hosts.

## Objetivo

Mover **todos** los archivos/carpetas ocultas (`.[!.]*`) de `/home/necro/` a `/home/necro/.hidden/`, creando symlinks de vuelta en `~` que apunten a `~/.hidden/.<nombre>`.

**Excepciones** (permanecen como dirs/archivos reales en `~`, NO se mueven):
- `.hidden` — carpeta destino de la migracion (evita recursion)
- `.cache` — caché de apps, cambia frecuentemente
- `.config` — configuracion de apps, ya gestionada por Home Manager
- `.local` — binarios locales, datos XDG
- `.nix-profile` — critico para nix
- `.nix-defexpr` — critico para nix
- `.dataSync` — carpeta de configuracion de Syncthing, si se mueve rompe Syncthing
- `.Trash-1000` — papelera del sistema
- `.steampath` — symlink de steam, rompe si se mueve
- `.steampid` — archivo PID de steam

## Archivos Creados

> **Nota:** Este proyecto implementa la estrategia **SYMLINK** (prioridad 3). Los dotfiles marcados como **HM** o **ENV** en el inventario deben migrarse a sus respectivos modulos/variables antes de ejecutar este proceso. Los dotfiles **SYMLINK** son los que el modulo y scripts siguientes mueven a `.hidden/`. Consultar la tabla en "Inventario de Dotfiles y Estrategia" para saber que hacer con cada uno.

### 1. `home-manager/modules/hidden-dotfiles.nix`

Modulo de Home Manager que hace dos cosas:

**A) `home.activation.moveHiddenFiles`** — Hook que se ejecuta en cada `home-manager switch`. Despues de que Home Manager crea sus propios symlinks (writeBoundary), itera sobre `.[!.]*` en `~`, y para cada uno:
- Si esta en la lista de excluidos → saltar
- Si ya es un symlink hacia `~/.hidden/` → saltar
- Si es un symlink a otro destino → no tocar
- Si `~/.hidden/.<nombre>` ya existe → eliminar el duplico en `~`
- Si no → `mv` a `~/.hidden/` y crear symlink `ln -sf`

**B) `systemd.user.services.hidden-dotfiles-watcher`** — Servicio systemd que ejecuta `watch-hidden-files.sh` como servicio persistente (Type=simple, Restart=on-failure). Se activa en el proximo login (`WantedBy=default.target`).

### 2. `home-manager/scripts/watch-hidden-files.sh`

Script bash que usa `inotifywait` (de `inotify-tools`) para vigilar `$HOME` en tiempo real. Cuando detecta un `create` o `moved_to` de un archivo/carpeta oculto en el raiz de `$HOME`:
- Verifica que no este excluido
- Verifica que no sea symlink existente
- Si ya existe en `.hidden` → eliminar duplico y crear symlink
- Si no → `mv` a `.hidden/` y crear symlink

Filtra con `--exclude '\.hidden/.*'` para no monitorear cambios dentro de `.hidden/`.

### 3. `home-manager/scripts/move-hidden-files.sh`

Script standalone identico al activation hook. Para testing manual sin pasar por Home Manager:
```bash
bash ~/nixFlake/home-manager/scripts/move-hidden-files.sh
```

### 4. Cambio en `home-manager/shared/default.nix`

Se agrego `../modules/hidden-dotfiles.nix` a la lista de imports.

## Inventario de Dotfiles y Estrategia

Cada dotfile en `~` tiene una de estas estrategias:

| Prioridad | Estrategia | Descripcion |
|---|---|---|
| 1 | **HM** | Migrar a modulo de Home Manager — desaparece de `~` |
| 2 | **ENV** | Redirigir con variable de entorno en `home.sessionVariables` — desaparece de `~` |
| 3 | **SYMLINK** | No tiene alternativa → se mueve a `.hidden/` con symlink |
| 4 | **EXCLUIDO** | No se toca por seguridad |

## Precauciones Importantes

### NO MOVER

La lista completa en **Excepciones** (seccion Objetivo). Detalles criticos:

- `.nix-profile` y `.nix-defexpr` — Si se mueven, `nix`, `nix-env`, `home-manager` y cualquier comando nix deja de funcionar. Son symlinks gestionados por nix.
- `.steampath` y `.steampid` — Steam los crea como symlinks internos. Moverlos rompe los juegos.
- `.config` — Home Manager crea symlinks dentro de `~/.config/` (fuzzel, kitty, zathura, waybar). Si `.config` fuera un symlink a `.hidden/.config`, los symlinks anidados crearian conflictos.
- `.hidden` — La carpeta destino de la migracion. Si no se excluye, el hook intentaria moverla dentro de si misma.

### COMPATIBILIDAD

Estos dotfiles **ya son symlinks de HM** a `/nix/store/` (no van a `.hidden/`). El hook los salta porque son symlinks a otro destino:
- `.bashrc`, `.bash_profile`, `.profile` — Gestionados por `programs.bash`
- `.gtkrc-2.0` — Gestionado por `gtk`
- `.Xresources` — Gestionado por `xresources`

Dotfiles **SYMLINK** (van a `.hidden/`) — verificacion de compatibilidad:
- `.ssh` — SSH sigue symlinks correctamente si se migra a `.hidden/`. Alternativa mejor: `programs.ssh`.
- `.steam/` — Steam sigue symlinks pero juegos con Proton pueden fallar. Probar antes de hacer permanente.
- `.pki/` — NSS (Chromium, Qt WebEngine) sigue symlinks.
- `.var/` — Flatpak hardcodea `~/.var`. Symlink funciona pero verificar apps Flatpak.
- `.steampath` y `.steampid` — Excluidos, no se tocan.

### INOTIFY-WATCHER
- Requiere `inotify-tools` en `home.packages`. Ya esta declarado en el modulo.
- El watcher monitorea `$HOME` y excluye `.hidden/` con `--exclude '\.hidden/.*'`.
- Puede consumir recursos si hay mucha actividad de archivos. `inotifywait -m` es liviano pero siempre corriendo.
- Se reinicia automaticamente si falla (`Restart=on-failure`, `RestartSec=5`).

### HOME MANAGER ACTIVATION
- El hook corre despues de `writeBoundary`. Esto es importante porque Home Manager ya creo sus propios symlinks (`.config/fuzzel`, `.config/kitty`, etc.) antes de que nuestra migracion se ejecute.
- Si un dotfile ya es symlink gestionado por Home Manager, nuestro hook lo detecta y lo salta.
- El hook es idempotente: ejecutarlo multiples veces produce el mismo resultado.

## Comandos de Prueba (Dry-Run)

Orden correcto de validacion:

1. **Primero** crear los archivos (modulo, scripts, imports) y configurar las variables HM/ENV.
2. **Luego** ejecutar las pruebas:

```bash
# Verificar sintaxis del flake completo
nix flake check --show-trace

# Construir NixOS (sin aplicar, solo verifica compilacion)
nixos-rebuild build --flake ~/nixFlake#desktop
nixos-rebuild build --flake ~/nixFlake#notebook

# Construir Home Manager (sin aplicar, solo verifica compilacion)
home-manager build --flake ~/nixFlake#"necro@desktop"
home-manager build --flake ~/nixFlake#"necro@notebook"
```

Si todo compila, proceder con `home-manager switch` para aplicar la migracion.

## Flujo de Activacion

```bash
# 1. Verificar sintaxis (dry-run)
nix flake check --show-trace
home-manager build --flake ~/nixFlake#"necro@$HOSTNAME"

# 2. Aplicar el modulo de Home Manager
home-manager switch --flake ~/nixFlake#"necro@$HOSTNAME"

# 3. El activation hook ejecuta la migracion automaticamente

# 4. Reiniciar sesion para que systemd active el watcher
# (o ejecutar manualmente)
systemctl --user start hidden-dotfiles-watcher.service
```

## Protocolo de Decision para Dotfiles Nuevos

Cuando aparezca un nuevo dotfile en `~`, seguir este orden:

1. **¿HM?** — ¿Existe un modulo de Home Manager que gestione el programa? Si si, configurarlo y el dotfile desaparece.
2. **¿ENV?** — ¿El programa soporta una variable de entorno XDG (XDG_CONFIG_HOME, XDG_DATA_HOME, XDG_STATE_HOME, XDG_CACHE_HOME) o una variable propia documentada? Si si, agregar a `home.sessionVariables`.
3. **¿SYMLINK?** — Sin alternativa → se mueve a `.hidden/`. Verificar que el programa siga symlinks.
4. **¿EXCLUIDO?** — ¿Es critico para el sistema (Nix, Steam, Syncthing, XDG bases)? Si si, agregar a la lista de excluidos.

## Que Hacer si Algo Falla

1. **Symlinks rotos**: `find ~ -maxdepth 1 -xtype l` — Symlinks que apuntan a destinos inexistentes. Solucion: mover manualmente el archivo de vuelta.
2. **Apps que no encuentran configs**: Verificar que la app sigue el symlink. Algunas apps hardcodean rutas absolutas y no resuelven symlinks. Si falla, devolver el dotfile a `~` y marcarlo como EXCLUIDO.
3. **Home Manager conflict**: Si HM crea un symlink en un path que ya es un symlink a `.hidden`, eliminar el symlink manualmente y re-ejecutar `home-manager switch`.
4. **Watcher no activo**: `systemctl --user status hidden-dotfiles-watcher` — Verificar que `inotify-tools` esta instalado.
5. **El hook movio algo que no debia**: `ls -la ~/.hidden/` — Si hay archivos que deberian estar excluidos, moverlos de vuelta y agregarlos a la lista de excepciones.
6. **Recursion**: Si `.hidden/` aparece dentro de `.hidden/`, se movio por error. El hook debe excluir `.hidden` explicitamente.
